# CS-255 System Analysis and Design

## DriverPass Project

The DriverPass system was designed for a small driving school business. The client wanted a web-based scheduling, training, and account-management system that would allow students to sign up for driving lessons, take online practice tests, and allow staff to manage operations securely and efficiently. 

Their existing processes were manual, inefficient, and error-prone. They wanted to modernize their business by putting it on a central digital platform. Through doing so, they optimized their processes and made them more efficient.

## Project Reflection

Reflecting on what I did well with this project, I would say that it was the analysis and design - that was what this module was about after all. With a keen eye for detail, I honed in on the client's requirements and accurately captured them in the included business requirements document.

If I could change anything about these documents though, it would be to enrich the detail on the various diagrams. ## How I interpreted the user's needs

I approached the DriverPass requirements by breaking the client’s statements into structured categories:
- Business goals → Improve scheduling accuracy, reduce manual work, expand online learning.
- User roles → Students, instructors, secretary, owner, and IT support.
- Choke points → Double‑booking, inconsistent communication, lack of online practice tests, no centralized data.
- Operational constraints → Limited staff, need for reliability, need for secure access control.

From these, I derived system requirements such as:
- Online student portal for registration, purchasing packages, scheduling lessons, and taking practice tests.
- Instructor scheduling module with real‑time availability and conflict prevention.
- Administrative dashboard for managing users, vehicles, packages, and reports.
- Role‑based access control to ensure each user only sees what they need.
- Cloud‑based architecture for availability, scalability, and remote access.

Each requirement directly traced back to a user need expressed in the client interview.

## Why Considering User Needs Is So Important in System Design

A system starts and ends with a user. It was conceived due to a need that arose and its implementation should end with meeting that need. So, designing with this in mind is critical because:
- It prevents building the wrong system. Without user‑driven requirements, features may not solve the actual business problems.
- It ensures usability. Systems fail when users find them confusing, inefficient, or misaligned with their workflow.
- It reduces rework and cost. Early alignment with user expectations avoids expensive redesigns later.
- It improves adoption. When users see their needs reflected in the system, they are more likely to embrace it.
- It supports long‑term scalability. Understanding real processes helps create a system that can grow with the business.

## Software Design Approach

My software design process begins with understanding the business problem and identifying what the system must accomplish. I analyze user roles, workflows, and pain points, then translate those findings into clear functional and non‑functional requirements. From there, I model the system using diagrams and structured documentation to validate logic, identify gaps, and ensure alignment with user expectations. The design is refined iteratively to improve clarity, usability, and maintainability.

## Techniques and Strategies for Future System Analysis and Design
Going forward, I would continue using the following strategies:
- User‑Centered Analysis: Prioritizing user needs, workflows, and constraints to ensure the system solves real problems.
- Model‑Driven Design: Using use case diagrams, activity diagrams, sequence diagrams, and class diagrams to visualize system behavior and structure.
- Incremental Refinement: Iteratively validating assumptions and updating requirements and models as understanding improves.
- Requirements Traceability: Maintaining clear links from user needs → requirements → design → implementation.
- Security‑First Thinking: Incorporating authentication, authorization, and data protection early in the design process.
- Clear Documentation: Producing structured, readable artifacts that support communication with stakeholders and developers.

These techniques help create systems that are scalable, maintainable, and aligned with both business goals and user expectations.
